5. 在程序入口处进行“组装” (Composition Root)

这是见证奇迹的时刻：你通过改变对象的组合方式，改变了系统的行为，而没有修改业务逻辑代码。

为什么在 Rust 中这样做非常牛逼？

  1. 零成本抽象 (Zero-cost Abstraction):
    如果你使用泛型（如上所示），Rust 编译器在编译时会进行“单态化”（Monomorphization）。它会直接生成一套专门调用 Sqlite 后接 Redis 的机器码。没有运行时的虚函数表查找开销，性能等同于你手写的硬编码代码。
  2. 职责单一 (SRP):
    • OrderService: 只管业务逻辑（价格对不对）。
    • SqliteRepo: 只管 SQL 怎么写。
    • CachedOrderRepo: 只管 Redis 怎么存。
      如果你下周要加 Kafka，你只需要再写一个 KafkaOrderRepo 并在 main 函数里再套一层。
  3. 极度好写测试:
    • 你想测 OrderService？给它塞一个 MockRepo（不连 DB，不连 Redis）。
    • 你想测 CachedOrderRepo 的缓存逻辑？给它塞一个 MockRepo 作为 inner。你可以精确测试“如果数据库写失败了，缓存还会不会写”这种边界情况。
  4. 符合开闭原则 (OCP):
    软件实体应对扩展开放，对修改关闭。你通过增加新类实现了新功能，而不是通过修改旧代码实现新功能。

总结

这种 impl<T: Trait> Trait for Wrapper<T> 的写法，是 Rust 高级架构的核心。你之前问的 impl<T: Runtime> TypedRuntime for T 也是类似的思想：基于已有的能力（Runtime/Inner），通过包装提供新的能力（Typed API/Caching），而不需要去动底层的代码。

这种“套娃”顺序的讲究

在组装时，你可以根据业务需求自由调整顺序，这体现了极大的灵活性：

  1. Retry 包裹 Cache (Retry(Cache(DB))):
    • 语义：如果“写DB+写缓存”整体失败了，就重试整个过程。
    • 适用场景：你希望保证缓存和数据库最终一致。
  2. Cache 包裹 Retry (Cache(Retry(DB))):
    • 语义：重试只针对 DB 写入。只有 DB 真的写成功了，才去写一次缓存。
    • 适用场景：你认为缓存写入通常不会失败，只有 DB 写入需要重试。
  3. Logging 包裹一切 (Log(Retry(Cache(DB)))):
    • 语义：记录最外层的请求和最终结果。

总结这种模式在 Rust 中的威力

  • 没有代码入侵：你为了加“重试”功能，只是写了一个新的 struct 实现了 trait。现有的核心业务逻辑（OrderService）完全不知道外面发生了什么，它甚至不知道自己被重试了。
  • 静态分发的高性能：由于 Rust 的泛型展开（Monomorphization），这段“套娃”代码在编译后会被优化成类似于： 没有任何运行时开销（不需要像 Java 那样通过反射或动态代理去寻找方法）。

这种“面向契约（Trait）”编程的思想，配合“装饰器模式”，能让你的 Rust 项目在变得极其复杂的同时，依然保持每一部分代码的纯粹和简单。
